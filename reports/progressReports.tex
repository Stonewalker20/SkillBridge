\section*{Final Report}
\addcontentsline{toc}{section}{Final Report}

\noindent\textbf{Report Focus:} Final system scope, implemented use cases, current architecture, updated screenshots and diagrams, and representative code from the completed SkillBridge platform.

\vspace{0.5em}
\noindent\textbf{Summary:}
By the end of the project, SkillBridge had grown into a full-stack portfolio and job-readiness platform that supports evidence-backed skills, resume ingestion, job-match analysis, admin moderation, analytics, personalization, help-request workflows, and a guided onboarding experience. The final build uses a React frontend, FastAPI backend, MongoDB persistence, and a normalized data model that keeps user-facing APIs stable while supporting more advanced analytics and moderation features behind the scenes.

\vspace{0.75em}
\noindent\textbf{What is Complete in the Final Build}
\begin{itemize}
  \item Authenticated account management with role-aware routing, password reset flows, and dedicated account subpages.
  \item Portfolio workflows for projects, evidence, GitHub link evidence submission, skill support, and dashboard summaries.
  \item Skill workflows for confirmations, manual proficiency, skill detail views, unsupported-skill detection, analytics, learning paths, and career paths.
  \item Job workflows for job ingestion, readable match analysis, moderation review, role tagging, and saved job history.
  \item Tailored-resume data generation and storage workflows, with saved tailored outputs and export-ready resume records.
  \item Admin workflows for user management, job moderation, taxonomy maintenance, MLflow support tooling, and help-request response management.
  \item User-experience workflows for onboarding guidance, achievements, workspace personalization, and unread support notifications.
\end{itemize}

\subsection*{Current Technical Architecture}
\addcontentsline{toc}{subsection}{Current Technical Architecture}
\begin{itemize}
  \item \textbf{Frontend:} React + Vite application with authenticated routes for Dashboard, Skills, Evidence, Job Match, Tailored Resumes, Analytics, Account subpages, Help, and Admin.
  \item \textbf{Backend:} FastAPI service exposing REST endpoints for auth, skills, confirmations, evidence, projects, portfolio, jobs, tailoring, dashboard analytics, taxonomy, rewards, help requests, and admin operations.
  \item \textbf{Database:} MongoDB collections for users, sessions, skills, evidence, projects, project-skill links, resume snapshots, confirmations, jobs, job ingests, job match runs, tailored resumes, help requests, rewards, and analytics state.
  \item \textbf{Inference Layer:} Local AI-assisted skill extraction and job matching with configurable inference preferences and compatibility fallbacks.
  \item \textbf{Data Safety Layer:} Compatibility-first ObjectId normalization, backfill scripts, moderation backfill logic, and index cleanup to reduce redundancy without breaking existing routes.
\end{itemize}

\noindent
\textbf{Sample 1 -- Linking a Skill to a Project and Returning Joined Skill Data}
\begin{TerminalBlock}
async def link_skill_to_project(project_id: str, payload: ProjectSkillLinkIn, user=Depends(require_user)):
    project_oid = ObjectId(project_id)
    skill_oid = ObjectId(payload.skill_id)

    if not await db["projects"].find_one({"_id": project_oid, **ref_query("user_id", user["_id"])}):
        raise HTTPException(status_code=404, detail="Project not found")
    if not await db["skills"].find_one({"_id": skill_oid}):
        raise HTTPException(status_code=404, detail="Skill not found")

    q = {"project_id": project_oid, "skill_id": skill_oid}
    existing = await db["project_skill_links"].find_one(q)
    if existing:
        return existing

    doc = {**q, "created_at": now_utc()}
    await db["project_skill_links"].insert_one(doc)
    return doc

pipeline = [
    {"$match": {"project_id": project_oid}},
    {"$lookup": {"from": "skills", "localField": "skill_id", "foreignField": "_id", "as": "skill"}},
    {"$unwind": {"path": "$skill", "preserveNullAndEmptyArrays": True}},
    {"$project": {"skill_name": "$skill.name", "skill_category": "$skill.category"}}
]
\end{TerminalBlock}
\noindent
This code shows two important final-system design choices. First, project-to-skill membership is stored in a dedicated \texttt{project\_skill\_links} collection instead of duplicating the relationship inside the project document. That keeps many-to-many relationships consistent and avoids duplicate skill arrays across project records. Second, the listing pipeline uses \texttt{\$lookup} and \texttt{\$project} so the frontend receives the skill name and category directly, which reduces extra round-trips from the client.

\vspace{0.75em}
\noindent
\textbf{Sample 2 -- Aggregating Dashboard Skill Support from Evidence}
\begin{TerminalBlock}
rows = await (
    db["evidence"]
    .aggregate(
        [
            {"$match": {"user_id": {"$in": user_refs}}},

for r in rows:
    sid = r.get("_id")
    skill = await db["skills"].find_one({"_id": {"$in": ref_values(sid)}}, {"name": 1, "category": 1})
    top_skills.append(
        {
            "skill_id": oid_str(sid),
            "skill_name": (skill or {}).get("name", ""),
            "category": (skill or {}).get("category", ""),
            "evidence_count": int(r.get("evidence_count", 0)),
        }
    )
\end{TerminalBlock}
\noindent

\subsection*{Week 1 -- How the UI Displays Data}

\begin{itemize}
    \item \textbf{Dashboard, The Overview:} The Dashboard acts as a command center, using individual summary widgets to display diverse data types: Key Metrics: Large-font text (e.g., "12/15 Skills") highlights critical progress. Lists: Simple bulleted lists display "Top Skills" and "Recommended Actions." Data Tables: The "Latest Job Analyses" section uses a traditional row-based table to compare specific data points like company names and match percentages.
    \item \textbf{Skill Catalog, Categorized Grids:} The "Skills" page presents information in a responsive grid of cards. Each card is standardized with: Category Tags: Colored labels (e.g., "Data Science") to define the skill's domain. Metadata Aliases: Smaller tags at the bottom of the card showing related search terms or versions (like "py" or "python3").
    \item \textbf{Evidence, Sequential Grids:} The Evidence section displays data as a vertical list of itemized cards. This allows for more detailed descriptions of specific files: Type Labels: Color-coded badges (Resume, Paper, Project) quickly identify the source of the data. Extracted Tags: Horizontal pill tags visualize the specific skills the system has pulled from each uploaded artifact.
    \item \textbf{Job Browser, Master-Detail Pattern:} The "Jobs" interface uses a split-pane layout common in productivity apps: Master List (Left): Concise preview cards showing the job title, company, and location. Detail View (Right): A larger space that expands when a card is selected, providing deeper data like specific skill requirements and an "Analyze Match" call-to-action.
\end{itemize}
